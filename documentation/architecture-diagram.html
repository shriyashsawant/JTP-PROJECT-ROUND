Plate 01

System Topology

Three containers on one Docker bridge network, address each other by service name (never localhost). Groq sits outside the network entirely — an optional HTTPS call, not a dependency.

Solid line — required, in-network call Dashed line — optional, external call Only frontend is meant to be public-facing in a hardened deployment; today all three ports are published for local review.
Plate 02

Request Lifecycle

One POST /api/v1/search/context call, start to finish. Every request passes through auth and rate-limiting before business logic ever runs; the LLM step is the only one that's skippable without changing the response shape.

always runs optional / skippable
Step 8 runs on the full candidate pool with cheap, numeric-only scoring; step 10 only builds the expensive explanation text for whatever survives truncation. That split exists because the ANN pool width was recently widened 5–20× — explaining every candidate before truncating used to be the slow path.

Data Model & Indexes

Two tables carry the whole system: the catalog, and the keys that gate access to it. Three index types on perfumes exist for three unrelated query shapes — nearest-neighbor, array containment, and fuzzy text.

Plate 04

Scoring & Diversity Pipeline

How a candidate pool of up to a thousand rows narrows to the handful actually shown — and why it narrows in this specific order.

Amber border — a diversity cap (shares one implementation, _cap_by_key) Dashed border — optional / conditional stage Gold border — a safety net added after a real under-fill bug was found live
The two strict caps (C, D) run before the LLM sees anything, deliberately not backfilled at that stage — relaxing the cap to fill a wide pool would let an over-represented brand or accord climb right back up before diversity ever mattered. The final cap (F) is the opposite: backfilled, because a response shorter than requested is a worse outcome than a little repetition.
